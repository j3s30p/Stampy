## Intent

바이브 코딩의 속도는 유지하되, 작성자가 바로 코드를 만지기 전에 _무엇을 성공으로 볼지_ 와 _어떤 계약을 건드리는지_ 를 짧게 고정한다. 본 skill 은 제안자 역할이다. 코드를 수정하지 않는다.

## Output Shape

작업 전 제안은 길게 쓰지 않는다. 아래 5개만 답한다.

1. 목표 — 한 문장.
2. 범위 — 만질 가능성이 큰 파일/영역.
3. 읽을 문서 — `docs/*.md` 또는 `skills/*/SKILL.md` 중 필요한 것만.
4. 위험 — invariant, ownership, 테스트 누락, UX/성능 중 실제 위험만.
5. 완료 조건 — 체크리스트 3~6개.

## Rules

- 구현 세부를 과하게 확정하지 않는다. 첫 제안은 방향과 경계를 정하는 용도다.
- `AGENTS.md` 의 critical invariant 를 반드시 대조한다.
- issue 가 있으면 issue 의 area label / acceptance criteria 를 우선한다.
- 하네스·문서 작업이면 `npm run harness:check` 를 완료 조건에 넣는다.
- 코드 작업이면 `npm run quality:fast` 를 완료 조건에 넣는다.
- UI 작업이면 실제 화면 확인 또는 스크린샷/영상 검증을 완료 조건에 넣는다.
- 사용자가 승인하거나 "바로 해" 라고 한 뒤에만 작성자 역할로 넘어간다.

## Anti-Patterns

- 사용자가 논의하자고 했는데 파일을 수정하는 것.
- “다음에 할 일”을 자동으로 새 파일에 남기는 것.
- 불확실한 외부 사실을 확인 없이 전제로 두는 것.
- 모든 docs/skills 를 미리 읽으라고 지시하는 것.
